# Science Guide

How Rubin Scout processes astronomical data, what cross-matching it performs, and how to interpret the results. Written for astronomers evaluating the tool's scientific utility, and for curious non-experts who want to understand what's happening under the hood.

## What Rubin Scout Is (and Isn't)

Rubin Scout is a **downstream visualization and cross-matching tool**. It does not classify alerts, run photometric pipelines, or process raw telescope data. It consumes transient discoveries from the IAU Transient Name Server (TNS), enriches them with machine learning classifications from ALeRCE, cross-matches them with astronomical catalogs, and provides gravitational wave counterpart search functionality.

Think of it as the layer between "a telescope detected something" and "a human understands what it means and can decide what to observe next."

## Data Flow

1. **TNS (Primary Source):** The IAU Transient Name Server is the authoritative registry for transient astronomical discoveries. TNS publishes daily CSV files containing all newly reported transients with spectroscopic classifications, precise coordinates, discovery dates, and redshifts. Rubin Scout ingests these daily files.

2. **ALeRCE (ML Enrichment):** For objects without spectroscopic classifications, Rubin Scout queries ALeRCE for machine learning classifications. ALeRCE processes ZTF alerts (and will process Rubin/LSST alerts) using stamp-based and light-curve-based classifiers. The system stores the full probability vector from ALeRCE's light-curve classifier, not just the top prediction.

3. **Light Curve Retrieval:** The system fetches photometric time-series data from:
   - TNS API (for objects with reported photometry)
   - ALeRCE API (for ZTF light curves with quality scores)

4. **SIMBAD Cross-Matching:** Every object is cross-matched against SIMBAD within 5 arcseconds to identify known astronomical associations (known galaxies, AGN, variable stars, etc.).

5. **Storage:** All enriched data is stored in PostgreSQL with PostGIS spatial indexing for efficient sky-position queries.

6. **API and Dashboard:** The processed, enriched data is served through a REST API and interactive web dashboard with human-friendly translations.

## Classification Sources and Confidence

Rubin Scout displays two types of classifications:

**Spectroscopic (TNS):** When TNS reports a spectroscopic type (e.g., "SN Ia"), this is a human-verified classification from actual spectrum analysis. These have `classification_probability = 1.0` (certain) and `classifier_name = "spectroscopic"`.

**Machine Learning (ALeRCE):** For objects without spectroscopic confirmation, ALeRCE provides probabilistic classifications from its light-curve classifier. The probability ranges from 0 to 1, and the full probability vector across all classes is stored and displayed.

The dashboard shows both the top classification and the full probability distribution, allowing users to assess confidence and identify ambiguous cases.

## Classification Taxonomy

Rubin Scout tracks the following transient classes:

| Class | Full Name | What It Is | Typical Timescale |
|-------|-----------|-----------|------------------|
| SNIa | Type Ia Supernova | White dwarf thermonuclear detonation | Weeks to months |
| SNII | Type II Supernova | Massive star core collapse (hydrogen-rich) | Weeks to months |
| SNIbc | Type Ib/c Supernova | Stripped-envelope core collapse | Weeks to months |
| SLSN | Superluminous Supernova | Extremely energetic explosion (10-100x normal SN) | Months |
| TDE | Tidal Disruption Event | Star torn apart by supermassive black hole | Months to years |
| KN | Kilonova | Neutron star merger (GW counterpart) | Days |
| AGN | Active Galactic Nucleus | Supermassive black hole accretion | Variable (stochastic) |
| Blazar | Blazar | AGN with jet pointed at Earth | Variable |
| QSO | Quasar | Luminous AGN at cosmological distance | Variable |
| CV/Nova | Nova | Thermonuclear flash on white dwarf surface | Days to weeks |

## Light Curve Quality: The Real/Bogus Score

Each ZTF detection includes an `rb` (Real/Bogus) score from 0 to 1, generated by a machine learning classifier that distinguishes real astrophysical sources from artifacts (cosmic rays, image processing glitches, satellite trails, etc.).

**Interpretation:**
- **rb > 0.55:** Likely real astronomical source
- **rb < 0.55:** Likely artifact or noise

This score is stored for all detections and can be used for quality filtering, though it's not currently exposed in the dashboard interface.

## Gravitational Wave Cross-Matching

This is Rubin Scout's unique scientific feature and its primary research contribution.

**How it works:** When a user selects a GW event, the system queries the database for optical transients that fall within the event's credible sky region and a configurable time window.

**Spatial matching:** For well-localized events (like GW170817 with 28 deg² at 90% credibility), the system approximates the credible region as a circle with radius r = sqrt(area_90 / π) and uses PostGIS ST_DWithin for efficient spatial queries from the skymap centroid.

**Time window:** Searches from 7 days before the GW event (to catch pre-existing transients that might be coincident) to 30 days after (configurable up to 365 days). The time window accounts for both:
- Pre-cursors (some models predict optical emission before merger)
- Post-merger emission (kilonova, afterglow, late-time components)

**Limitations of the current approach:**
- **Circular approximation:** Uses skymap centroid + circular radius rather than the full HEALPix probability map. This simplification works well for compact skymaps (GW170817) but is less accurate for large, irregular, or multi-modal distributions.
- **No luminosity distance filtering:** Does not incorporate the GW posterior on luminosity distance. A future version could filter candidates by comparing host galaxy redshift to the GW distance estimate.
- **Uniform probability assumption:** All candidates within the credible region are treated equally; the actual probability density within the region is not used for ranking.
- **No rate/luminosity priors:** Does not weight candidates by expected kilonova luminosity, supernova rates at the inferred distance, or other astrophysical priors.

**Planned improvements:**
- Full HEALPix skymap integration
- Luminosity distance filtering using host galaxy redshifts
- Bayesian ranking using source rate priors
- Integration with telescope scheduling systems

**Pre-loaded events:** Six notable GW events from GWTC catalogs are included with human descriptions: GW170817 (first BNS with EM counterpart AT2017gfo), GW190425 (second BNS), GW190521 (most massive BBH), GW200105 and GW200115 (first NSBH detections), and GW231123 (highest-mass BBH in GWTC-4.0).

## SIMBAD Cross-Matching

For each ingested object, Rubin Scout queries SIMBAD within 5 arcseconds. This identifies whether the transient position coincides with a known astronomical object (galaxy, star, AGN, etc.).

**What the match distance means:** 5 arcseconds is a generous radius for point-source matching at ZTF's ~2 arcsecond resolution. In crowded fields (galactic plane, dense clusters), spurious matches are possible. The match distance is always stored and displayed, allowing users to assess the quality of the association.

**Match interpretation:**
- **< 1 arcsecond:** Very likely physical association
- **1-3 arcseconds:** Probable association
- **3-5 arcseconds:** Possible association, verify visually
- **Host galaxy matches:** Many transients occur in known galaxies; a match to a galaxy catalog entry is expected for extragalactic transients

**Column name handling:** SIMBAD recently changed their API column names (MAIN_ID to main_id, etc.). Rubin Scout handles both formats for backward compatibility.

## Coordinate Systems

- All sky positions stored in **ICRS (J2000)**, the standard astronomical reference frame
- RA in degrees (0 to 360), Dec in degrees (-90 to +90)
- PostGIS geography column for proper great-circle distance calculations
- Cone search radius specified in arcseconds
- Constellation boundaries use simplified rectangular approximations (see limitations below)

## Time Conventions

- All timestamps stored as timezone-aware UTC
- Modified Julian Date (MJD) preserved alongside converted timestamps for compatibility with astronomical tools
- MJD-to-UTC conversion uses Astropy's Time class (MJD = days since November 17, 1858)
- Dashboard displays relative time ("3 days ago") for readability
- API accepts and returns ISO 8601 formatted timestamps

## The Cosmos Translation Layer

Rubin Scout's `cosmos.js` module maps every classification to:
- A human-readable name and one-line summary
- An emoji identifier for quick visual scanning
- A full paragraph explaining what the event type is and why it matters
- An approximate constellation name derived from RA/Dec coordinates

**This is not "dumbing down" the science.** It's making the same information accessible to multiple audiences:
- Professional astronomers can use the API for raw data
- Graduate students can use the dashboard for quick assessment
- Undergraduate students can explore without astronomical jargon barriers
- Curious non-experts can understand what telescopes are discovering

The raw data (RA/Dec, MJD, magnitude, classifier probabilities) is always available through the API and detail pages for those who need it.

## Limitations and Caveats

1. **Classification accuracy depends on upstream sources.** Rubin Scout does not reclassify. It trusts TNS spectroscopic classifications (which are human-verified) and displays ALeRCE ML classifications with full probability distributions. If ALeRCE misclassifies an object, that error propagates.

2. **SIMBAD matching is positional only.** A 5-arcsecond match means "there's a known object near this position," not necessarily a physical association. Users should verify plausible associations using the match distance and object types.

3. **GW cross-matching is approximate.** The circular skymap approximation and uniform probability assumption are simplifications of the real probability distribution. See the GW section above for detailed limitations.

4. **Light curves are stored as-is.** No host-galaxy subtraction, extinction correction, or photometric recalibration is applied. These are tasks for dedicated photometry pipelines, not downstream visualization tools.

5. **Constellation mapping is approximate.** Uses simplified rectangular boundaries for quick position-to-constellation lookup, not the official IAU boundaries. Accurate for "it's in the direction of Orion" context, not for precise boundary determination.

6. **Ingestion is polling-based (15-minute intervals).** The current system polls TNS daily and ALeRCE every 15 minutes via REST APIs. This is not real-time. The planned Kafka consumer will reduce latency to seconds for ALeRCE alerts.

7. **TNS coverage is authoritative but incomplete.** TNS contains only discoveries reported by observing teams. Not all transients discovered by ZTF or other surveys are reported to TNS. For comprehensive ZTF coverage, direct broker access (via ALeRCE Kafka streams or Pitt-Google Pub/Sub) would be required.
